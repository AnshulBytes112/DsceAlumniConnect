# Resume Parser Evaluation Report

This document explains the evaluation workflow used to measure the resume parser against the Kaggle dataset `snehaanbhawal/resume-dataset`.

The actual run artifacts are generated by `evaluate_resume_parser.py` and written to the output directory you pass with `--output-dir`.

## Purpose

The goal is to answer two practical questions:

1. How many resumes can be parsed before the configured Gemini API key is rate-limited or exhausted?
2. How complete is each parse result when mapped to the frontend-facing resume schema?

## Why the Metric Is a Proxy

The Kaggle resume dataset is commonly distributed as raw resume text and category labels, not as ground-truth structured JSON with validated field-by-field answers.

That means there is no honest exact-match accuracy score unless the dataset exposes a labeled schema. For that reason, the evaluator reports:

- Parse success rate
- Average structural completeness
- Effective efficiency

If the dataset contains labeled fields, the evaluator can be extended to add exact-match scoring later.

## What the Script Does

`evaluate_resume_parser.py` performs the following steps:

1. Downloads the Kaggle dataset with `kagglehub`.
2. Auto-detects a usable resume source.
3. Uses the same resume JSON schema as the backend parser, including `graduationYear` and `department`.
4. Sends each sample to the Gemini API using the provided API key.
5. Normalizes the response into the backend DTO shape.
6. Scores the result for completeness.
7. Stops early if a quota or rate-limit error is encountered.
8. Writes `summary.json`, `results.json`, and `report.md`.

## Output Files

The evaluator writes these artifacts:

- `summary.json`: aggregate metrics and run metadata
- `results.json`: per-sample parse outcomes and scores
- `report.md`: human-readable evaluation report

## Metric Definitions

### Parse Success Rate

The percentage of samples that returned valid structured JSON and normalized successfully.

### Average Completeness

The average schema coverage across:

- Profile
- Work experience
- Education
- Projects
- Achievements
- Skills

### Effective Efficiency

Defined as:

`parse success rate * average completeness`

This gives a single practical score for how well the API key and prompt are performing together.

## Run Instructions

Install dependencies:

```bash
pip install -r backend/scripts/requirements-resume-eval.txt
```

Run the evaluator:

```bash
python backend/scripts/evaluate_resume_parser.py --max-samples 50 --output-dir backend/scripts/reports/resume-parser-eval
```

If you want to use a specific Kaggle file path, add `--file-path`.

## Suggested Interpretation

- High parse success + high completeness: the current prompt and API key are working well.
- High parse success + low completeness: the parser needs better post-processing or schema tuning.
- Low parse success + quota errors: the API key or model quotas need adjustment.
- Low parse success + no quota errors: the prompt, dataset format, or request handling needs review.

## Recommendations

1. Keep the evaluation sample size small enough to control API spend.
2. Cache parsed resumes by content hash to avoid duplicate requests.
3. Add labeled-field scoring if the dataset exposes ground-truth annotations.
4. Keep `graduationYear` and `department` in the normalized profile contract so the frontend can consume them directly.
